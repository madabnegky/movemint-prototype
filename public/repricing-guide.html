1. What this tool solves

Movemint is moving from flat-rate annual fees to a hybrid pricing model: a SaaS base fee for platform access, plus a transaction fee on the activity Movemint drives (events generated, applications submitted, redemptions completed, or basis points on funded volume).

The Repricing Tool answers two questions for the existing book:

  1. For each existing client, what would the new pricing need to charge to land at 100–120% of their current revenue while keeping 60–80% of revenue from SaaS?
  2. Across the whole book, what's the publishable price per asset tier — the SaaS fee + per-event price (or BPS rate) we'd put on the rate card?
Strategic targets

Revenue mix: 60–80% SaaS / 20–40% transaction (per client)
Per-client revenue: 100–120% of 2025 actual

The tool back-solves both the recommended SaaS fee and the per-event/BPS price simultaneously for each combination of revenue target × pricing model, so you can compare options and pick the one that produces the best publishable price card.

2. Key concepts (read once, then skim later)

The four billing options

Every client will land on exactly one of these four pricing models. The Repricing tool computes recommended prices for all four so you can compare side by side.

OptionWhat's billedBest when…
BPS take-rateFloor + bps × funded loan / new deposit volumeVolume is high, predictable, and Movemint takes a cut at funding
Per-redemptionSaaS + $ per loan funded / account openedAligns price with realized customer value
Per-applicationSaaS + $ per application startedSpreads risk earlier in the funnel; pre-funding billing
Per-offer-generatedSaaS + $ per offer rendered to a memberHeavy member base, low conversion expectations (impression model)

Module configuration: every client is one of three

Each client uses one of these module configurations, inferred automatically from the event counts in your CSV:

  • Lending-only — All deposit-side event counts are 0 in the CSV. Client uses Movemint for loan-side originations only.
  • Deposit-only — All lending-side event counts are 0. Client uses Movemint for deposit growth only.
  • Both modules — Has activity on both sides.
  • No activity — All counts are 0. Edge case; will show in the roster but back-solve outputs will be sparse.

For both-modules clients, the tool can charge a different per-event price (or per-module bps rate) for lending vs. deposit. The default is lending priced 2× deposit (the "lending premium"), reflecting that a funded loan is typically more valuable than a new deposit account. The premium applies to both per-event prices and BPS rates.

The five revenue targets

For every client, the tool computes pricing at five revenue targets relative to their 2025 actual fee:

100%, 105%, 110%, 115%, 120%

You'll see a 5×4 grid (revenue target × billing option) on the per-client view. The tier summary uses the same targets but aggregates across clients.

The seven asset tiers

Tiers are based on credit-union total assets and used for grouping in the tier summary view. Boundaries are fixed; only prices recommended at each tier are computed by the tool.

  1. $0 – $500M
  2. $500M – $1B
  3. $1B – $2B
  4. $2B – $3B
  5. $3B – $4B
  6. $4B – $5B
  7. $5B+

The Repricing tool needs one row per existing client. Required columns (case-sensitive headers):

Tip

Use the "Download sample CSV" button on the Repricing tab to grab a starter file with the right columns and example rows. Match the column names exactly.

Numbers can include $ and commas

The parser strips $ and , from numeric fields, so $52,000 works the same as 52000.

Module mode is inferred — don't add a column for it

If a client uses lending only, set their three deposit columns to 0. The tool detects this and treats them as Lending-only. Same logic for Deposit-only. Don't add a "module" or "mode" column — it'd be redundant.

4. Step 2 — Upload and review the roster

1

Click "Upload CSV"

The button is in the upload card at the top of the Repricing tab. Pick your CSV file. The tool parses immediately; results appear below.

2

Read the validation feedback

Errors (red) — block individual rows from being imported. Most common: missing required columns, missing client name. Fix the CSV and re-upload.

Warnings (amber) — accept the row but flag a non-fatal issue. The most common warning: cu_number not found in NCUA data. The row still imports; you'll see "unmatched (XXX)" in the NCUA column of the roster table. Click that link to manually associate the client with an NCUA charter.

3

Scan the roster for sanity

The roster table shows: client name (clickable to drill in), NCUA match (or "associate…" link if unmatched), assets, asset tier, module mode badge, 2025 fee, and total event count.

Module mode is the colored badge: Lending-only, Deposit-only, Both modules, No activity. If a client looks like the wrong mode, double-check that you put the right zeros in the right columns.

4

Manage the roster

Delete one — click the trash icon at the end of any row.

Clear all — button next to upload. Wipes localStorage. Confirms before deleting.

Export — "Export JSON" button downloads the current roster as a JSON file. Useful for committing a known-good dataset to the repo, or sharing with a teammate.

How data persists

Your uploaded clients live in this browser's localStorage. They survive page reloads, but they don't sync to other browsers or other team members. To share, use "Export JSON" and have the recipient drop the JSON's contents back through the same upload flow (or commit it to the repo). To start fresh, hit "Clear all".

5. Step 3 — Tune the global controls

Two controls at the top of the Repricing tab affect every back-solve in the tool:

Target SaaS share (default 70%)

The percentage of total client revenue that should come from the SaaS base fee (or floor for BPS). The remainder goes to per-event / BPS revenue.

Allowed range: 50–90%. The "sweet spot" is 60–80%, matching the strategic target.

  • Higher (e.g. 80%) — More predictable revenue, less exposure to client activity. Per-event prices come out lower.
  • Lower (e.g. 60%) — More activity-aligned. Per-event prices come out higher; clients pay more when they use the platform.

Lending premium (default 2×)

For both-modules clients, this multiplier sets the ratio of lending event price to deposit event price. Loan conversions are typically more valuable than deposit account openings, so they're priced higher per event.

Allowed range: 1× (equal) to 5× (lending events priced 5× deposit).

Example with 800 lending + 400 deposit redemptions, $33K event revenue target:

PremiumLoan priceDeposit priceTotal event rev
1×$27.50$27.50$33,000
2×$33.00$16.50$33,000
3×$35.36$11.79$33,000

The premium just shifts how the same $33K is split between modules — total revenue always lands at target.

Lending-only / Deposit-only clients

The lending premium only matters for both-modules clients. Single-module clients have all event revenue allocated to the one module they use, regardless of premium.

BPS-specific assumptions (in the Assumptions modal)

Click Assumptions in the top-right of the Repricing tab to open a modal with the rest of the BPS knobs:

  • Loan funding rate (default 35%) — % of loan applications that result in a funded loan. Industry typical: 30–40%.
  • Deposit funding rate (default 60%) — % of deposit applications that result in a new account. Higher than loans because there's less underwriting friction.
  • Avg funded loan size (default $35K) — blended average across all loan types. Tune higher for mortgage-heavy books, lower for mostly-consumer books.
  • Avg new deposit account size (default $5K) — average balance of a newly-opened deposit account.

These four values feed the BPS back-solve. Funded volume is computed from the client's actual application counts in the CSV, multiplied by the funding rate, multiplied by the avg size — no NCUA dependency.

6. Step 4 — Drill into a specific client

Click any client's name in the roster to open their detail view. You'll see:

Stat strip

2025 annual fee, asset tier, total assets, and member count (the last three pulled from NCUA data).

Event count strip

Lending vs deposit counts for redemptions / applications / offers generated. The unused side is struck through for single-module clients, so you can see at a glance which side the back-solve is using.

The 5×4 grid

The headline output. Rows are revenue targets (100/105/110/115/120%); columns are the four billing options.

Each cell shows the recommended pricing for that combination:

  • BPS cells: $X/mo floor + Y bps loan / Z bps deposit applied to funded volume. Funded volume is derived from the client's applications + the funding rate / avg size assumptions.
  • Per-event cells: $X/mo SaaS + $Y per event (lending and deposit shown separately for both-modules clients).
  • Cells highlight green when feasible; grey when there's no event volume / no NCUA data.
How to read it

Read across a row to compare what each billing option needs to charge to hit the same revenue target. Read down a column to see how a single billing option scales as the revenue target rises.

7. Step 5 — Use the Tier Summary to set publishable pricing

From the client list, click "View tier summary →" to switch into the aggregate view. This is the deliverable for your team — what you'd put on the rate card.

Pick a billing option

Four toggle pills at the top: BPS take-rate / Per-redemption / Per-application / Per-offer-generated. Click one. Every tier's tables re-render to show pricing for just that billing option.

Three columns per tier

Each asset tier shows three sub-columns: Lending-only · Deposit-only · Both modules. Each column is populated from the subset of clients in that tier with that mode.

Each sub-column has a 5-row table:

  • Target — revenue target (100/105/110/115/120%)
  • SaaS / mo (or Floor / mo for BPS) — the median recommended monthly fee across clients in this tier+mode, with the min–max range below
  • Per-event price (or BPS rates per module for BPS) — the median recommendation

How to use it

  1. Pick a revenue target (e.g. 110% — what we want to land at on average)
  2. Pick a billing option (e.g. Per-redemption — leadership wants this model)
  3. Read the median price for each tier+mode
  4. That's your rate card: "Tier 3 lending-only clients pay $X/mo + $Y/redemption"
The deliverable

The tier summary view is what you take to your team to make the pricing decision. Switch billing options to compare; the column with the most stable recommendations across tiers (tightest min-max range) is usually the one to pick.

8. Methodology — what the math is doing under the hood

The basic identity

For every client, every revenue target, every billing option, the tool solves the same identity:

total_rev = saas_rev + event_rev
saas_rev = target_saas_share × total_rev
event_rev = (1 - target_saas_share) × total_rev
# where target_saas_share defaults to 70%

Given a target total revenue (e.g. $66K = 110% of $60K current fee) and a target SaaS share (e.g. 70%), the SaaS revenue is fully determined: $66K × 70% = $46.2K/yr → $3,850/mo.

The event revenue is then $66K × 30% = $19.8K/yr, divided across the client's actual event counts to get a per-event price.

Module split for both-modules clients

If a client has 800 lending + 400 deposit redemptions and the lending premium is 2×, we solve:

event_rev = price_loan × 800 + price_dep × 400
price_loan = 2 × price_dep # lending premium constraint

=> price_dep = event_rev / (2 × 800 + 400)
=> price_loan = 2 × price_dep

Both prices fall out cleanly. The total event revenue is preserved exactly; only the split between modules changes with the premium.

BPS back-solve

BPS produces two clean rate-card numbers per client (or per tier+mode in the tier summary): a recommended monthly floor and a recommended bps rate per module. There is no dependency on the BPS tab or NCUA balance-sheet data — funded volume is derived from the client's actual application counts plus a few global assumptions.

Step 1: derive funded volume from applications

loan_funded_volume = applications_lending × loan_funding_rate × avg_loan_size
deposit_funded_volume = applications_deposit × deposit_funding_rate × avg_deposit_size

Defaults: loan funding rate 35%, deposit funding rate 60%, avg loan size $35K, avg deposit size $5K. All are editable in the Assumptions modal at the top of the Repricing tab.

Step 2: back-solve to the strategic mix

recommended_floor = target_saas_share × target_rev / 12
target_bps_rev = (1 - target_saas_share) × target_rev

# lending-premium constraint (loan_bps = p × deposit_bps):
deposit_bps × (p × loan_volume + deposit_volume) = target_bps_rev
=> deposit_bps_rate = target_bps_rev / (p × loan_volume + deposit_volume)
=> loan_bps_rate = p × deposit_bps_rate
=> recommended_loan_bps = loan_bps_rate × 10000
=> recommended_dep_bps = deposit_bps_rate × 10000

The recommendation is two clean numbers — for example "12 bps loan / 4 bps deposit + $5,500/mo floor" — directly publishable on a rate card.

Mode filtering

Before back-solving, the tool zeroes out the unused module's counts:

  • Lending-only: deposit event counts and deposit applications set to 0. Per-event price (and deposit bps) floats up because revenue concentrates on the lending side.
  • Deposit-only: same, mirror image.
  • Both: no filtering; full event counts and volumes used.

This means a Lending-only client with 800 lending events at 110% target will get a higher per-event price than a both-modules client with the same 800 lending events plus 400 deposit events — because the both-modules client has more events to spread the same $19.8K event revenue across.

Tier aggregation

The tier summary takes every client back-solve in a tier+mode and computes the median across them. The min and max are also surfaced as a "spread" beneath the median — that tells you how varied the recommendations are within a tier.

Tight spread = the tier is pricing-coherent (you can publish one price). Wide spread = the tier has heterogeneous clients and a single tier price will help-some-hurt-others.

What's NOT in the back-solve

  • Tier prices from the prospective tabs — the Repricing tab does not read the tier prices set on the SaaS+event tabs. SaaS is an output here, not an input.
  • Floor from the BPS tab — same. The Repricing tab recommends a floor; it doesn't consume the one set elsewhere.
  • NCUA data is only used for asset-tier grouping — the per-event AND BPS back-solves only need the client's actual event/application counts, 2025 fee, and the global assumptions (funding rates, avg sizes). NCUA's only role is mapping a client to an asset tier in the tier summary view.

9. Common pitfalls and gotchas

Pitfall 1

Mode looks wrong on the roster. Module mode is inferred from event counts. If a client looks like Lending-only but should be Both modules, check that the deposit columns aren't all zero in the CSV. The tool can't know the client's intended mode — only what their data says.

Pitfall 2

BPS shows "no funded volume" for a client. Means both their lending and deposit application counts are 0 in the CSV — the BPS calc has nothing to apply rates to. If the client really does fund loans/accounts but the CSV captured no application data, the BPS column will stay grey for them. (No NCUA dependency for BPS — only for asset-tier grouping in the tier summary.)

Pitfall 3

Per-event prices look implausibly high or low. Two common causes: (1) the annual_fee_2025 is wrong in the CSV — usually a missing zero or a comma issue; (2) the event counts are off by an order of magnitude. Open the per-client detail view; the stat strip shows the parsed annual fee and event counts.

Pitfall 4

The tier summary shows huge spreads in some tier+mode cells. Means the clients in that segment have very different volume profiles. The recommended fix is to look at the per-client view for those clients and decide whether to (a) pick a price closer to the high end (and let some clients overshoot 120%), (b) pick the low end (and accept some undershoot 100%), or (c) propose two sub-tiers. The tool surfaces the spread; the strategic call is on you.

Pitfall 5

Multiple browsers / devices show different rosters. Client data lives in localStorage on whichever browser uploaded the CSV. Use "Export JSON" + share if you need consistency across the team.

Reset to defaults

If you want to start completely fresh: hit "Clear all" on the upload card, refresh the page (resets the global controls back to 70% SaaS / 2× lending premium), and re-upload your CSV.
